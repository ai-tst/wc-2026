# Регресс-тесты «Отсоса»

Лёгкий набор регресс-тестов без внешних зависимостей (на боксе нет pytest, и он
не нужен). Гоняем **перед каждым релизом** — если что-то красное, не катим.

## Запуск

```bash
# Полный набор: логика (Python+JS) + smoke по локальному серверу
bash tests/run.sh

# Только чистая логика — живой сервер не нужен
bash tests/run.sh --logic

# Smoke по проду вместо localhost
BASE_URL=https://51.250.35.235.sslip.io bash tests/run.sh
```

Код возврата `0` — всё зелёное. Любой другой — есть провал, релиз стопаем.

## Что покрыто

| Файл | Что проверяет |
|------|---------------|
| `test_scoring.py` | Серверный расчёт очков `_calc_match_points` и матчинг игроков `_players_match`/`_norm_player` (это логика TG-сообщений о результате). |
| `test_points.mjs` | Клиентский расчёт очков `public/points.js`: очки за матч, ауткрайты, раунды плей-офф, бонусы за стадию. Это то, что реально видит юзер в таблице. |
| `test_smoke.py` | Живой сервер: страница и статика отдаются, публичные API живы и не сломали JSON-контракт, приватные эндпоинты по-прежнему закрыты авторизацией. |

## Важно про паритет логики

Логика начисления очков продублирована: `server.py` (`_calc_match_points`, для
TG) и `public/points.js` (для таблицы). **Golden-значения в `test_scoring.py` и
`test_points.mjs` намеренно одинаковые.** Меняешь правила начисления — правь
**оба** файла кода и **оба** теста, иначе TG-сообщения и таблица разойдутся.

## Заметки по реализации

- `WC2026_TESTING=1` отключает в `server.py` побочки импорта (init_db,
  TG-тред, preload рейтингов) — чтобы юнит-тесты импортировали чистую логику без
  сети и записи в БД. В проде (gunicorn) переменная не выставлена → поведение
  не меняется.
- `public/package.json` (`{"type":"module"}`) нужен, чтобы Node импортировал
  `public/*.js` как ES-модули (в браузере они и так грузятся через
  `<script type="module">`).
- Smoke-тесты строго **read-only** — в прод-БД ничего не пишут.
- `_harness.py` — крошечный self-made раннер ассертов (вместо pytest).
